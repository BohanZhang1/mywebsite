## CSCI 5030: Principles of Software Development
**Saint Louis University, Fall 2026**

Three invited guest lectures delivered to a graduate-level software engineering course.

---

### Lecture 1: Test-Driven Development (TDD)

An in-depth lecture and live coding session on Test-Driven Development, covering the philosophy, mechanics, and industry context of writing tests before writing code.

**Topics covered:**

- The RED → GREEN → REFACTOR cycle and why each phase matters
- Why the failing test (RED) phase is intentionally the most important step
- Writing minimal code in the GREEN phase to avoid over-engineering
- When and how to safely refactor under a passing test suite
- How tests encode requirements and serve as living documentation
- TDD as a *design technique*, not merely a testing technique
- Test coverage: what it measures, why high coverage does not guarantee correctness
- TDD in the context of Extreme Programming (XP) and modern agile practices

---

### Lecture 2: Version Control: GitHub Merge Workflows and Conflict Resolution (Workshop)

A hands-on workshop on collaborative version control using Git and GitHub, with focus on the merge workflows that matter most in team settings.

**Topics covered:**

- Branching models: feature branches, GitFlow, and trunk-based development
- Pull request reviews and merge strategies (merge commit, squash, rebase)
- Understanding and resolving merge conflicts step-by-step
- Live workshop: students resolved real merge conflicts in a shared repository
- Best practices for commit messages and clean history

---

### Lecture 3: Unit Testing and TDD Workshop

A practical workshop reinforcing TDD concepts through guided implementation exercises.

**Topics covered:**

- Unit testing fundamentals: what to test, what not to test
- Test isolation and the role of mocks and stubs
- Writing testable code: dependency injection and interface design
- Workshop: students incrementally implemented a feature using the RED → GREEN → REFACTOR cycle
- Connecting unit testing discipline to long-term codebase maintainability

### Impact on Students' Thinking

When asked *"Did the workshop change how you think about writing code?"*, 9 out of 10 respondents answered **Yes**. Representative responses:

> "Yes, it reversed my approach. I used to write most of the implementation first, then test afterward. But now I understand how starting with tests reduces workload."

> "Yes, I was exposed to another philosophy of writing code and realized there is another way of reaching the same goal that may be better in certain situations."

> "Yes, I hadn't thought about writing test cases before coding so that I don't have a bias in how I think a successful run looks when I get to testing."

> "Yes, the workshop promotes a different way of thinking in at least three ways: (1) from code-first to tests-define-behavior-first, (2) from big batches of changes to small safe iterations, (3) from perfect code now to working, test-backed code first."
